Serverless 入门

2023-06-25 | Cloud Native

一提到 Serverless，经常会有两种极端的意见。

一种是觉得这玩意儿很神：不用管服务器，按量付费，自动扩缩容，听上去像云计算世界的终极答案。

另一种则更直接：名字都叫 Serverless 了，结果底层明明还是有服务器，这不就是个营销词？

这两种理解都有其道理，但都只看到了表层。

Serverless 真正有意思的地方，不在于“有没有服务器”，而在于：它改变了开发者和基础设施打交道的方式。

对于大多数业务团队来说，重点不再是我要申请多少机器、配多大规格、怎么扩容、怎么扛峰值，而是把代码和业务逻辑交出去，让平台按请求来接住它。

什么是 Serverless？

先说一句最不绕弯的话：

Serverless 不是没有服务器，而是你不再以服务器为核心来组织应用。

从执行、部署到计费，Serverless 提供的是一种更“托管化”的模型：

  • 资源按需分配
  • 基础设施由平台接管
  • 计费按实际调用发生
  • 应用可以随着流量自动扩缩容

也就是说，你不需要再把主要精力放在机器和容量规划上，而是更多关注代码本身。

狭义和广义的 Serverless

如果按行业里常见的说法来分，Serverless 大致有两个层面。

狭义 Serverless

通常指：

  • FaaS（Function as a Service）
  • BaaS（Backend as a Service）

前者让你把业务能力拆成函数，后者把数据库、鉴权、存储、消息等基础能力服务化。

广义 Serverless

它更像一种架构思想：

在构建和运行软件时，尽量不把“管理服务器”作为开发团队的核心负担。

这也是为什么今天很多容器平台、边缘平台、托管工作流，虽然不一定都长着同一张脸，但都多少沾点 Serverless 的味道。

Serverless 和传统架构，到底差在哪？

如果你以前主要做的是传统 Web 服务，那理解 Serverless 最快的方法，就是拿它和熟悉的架构做对照。

1. 资源分配方式不同

传统架构里，你要先准备资源，再承接流量。

Serverless 则反过来：先有请求，再由平台按需分配资源。

这件事的直接影响是，开发者不再需要对“机器规模”做过多前置判断。

2. 计费逻辑不同

传统架构按机器、实例、包年包月或固定规格收费比较常见。哪怕流量不大，资源也常常得一直挂着。

Serverless 更强调按量付费，粒度甚至可以细到毫秒级。真正有调用才有费用，没有请求时，理论上甚至可以缩到 0。

3. 弹性能力不同

传统应用当然也能扩缩容，但那往往需要你自己搭一整套机制，或者至少得配置清楚监控、阈值、策略和实例池。

Serverless 把这套复杂性尽量往平台层压。业务方只需要描述应用，扩缩容这件事由平台兜底。

4. 应用拆分方式不同

Serverless 更强调函数化、事件驱动和无状态。一个业务不一定是一整坨长期常驻服务，而可能被拆成多个对事件作出响应的函数单元。

为什么这几年大家都在谈 Serverless？

因为它确实解决了一些老问题，而且解决得很直接。

Serverless 的优势，不只是“省机器”

第一，团队不用把精力浪费在一堆底层脏活上

容错、容灾、自动扩容、日志、运行环境托管，这些事情在传统模式下都要有人操心。Serverless 的价值之一，就是尽量把这些事情平台化。

这并不意味着运维消失了，而是说很多重复劳动可以被统一抽走。

第二，弹性是真的更自然

很多业务最大的成本，不是高并发本身，而是你必须为了偶发高峰长期预留资源。

Serverless 的优势在于：高峰来了能扩，没流量时又能缩，资源利用率明显更高。

第三，试错成本更低

无论是小团队做新产品，还是成熟业务做新功能，Serverless 都很适合拿来快速试验。

你不必先搭出一整套重型基础设施，很多时候只需要先把核心逻辑跑起来。

第四，更适合事件驱动型系统

凡是“来一个事件，处理一次”的场景，Serverless 都天然合拍。它不要求应用必须像传统服务那样一直待命。

什么场景适合用 Serverless？

Serverless 不是万能药，但它非常适合一批典型场景。

比如：

  • Web 应用
  • 小程序后端
  • 物联网事件处理
  • 多媒体任务，比如转码、图片处理
  • 数据实时分析
  • 批处理异步任务
  • 定时任务
  • API 代理层
  • 边缘计算场景
  • 服务编排

一个很典型的例子：图片处理

以微博这类高并发图片场景为例，用户上传原图后，平台并不一定提前生成所有规格图片，而是可以在请求到来时，通过函数从对象存储拉取原图，再按不同终端动态加工并返回。

这种模式特别适合波峰波谷明显、处理逻辑相对独立的业务。

再比如实时协作和内容转换

像石墨文档那类多人实时编辑场景，流量峰值往往极其短促，但对延迟又非常敏感。Serverless 可以把部分处理逻辑放到函数里，让平台去接住瞬时波峰。

而语雀这类内容转换场景，则很适合把高风险、CPU 密集或依赖复杂第三方软件的处理逻辑放入函数隔离执行。这样既能把风险隔开，也能避免长期养一组大集群。

Serverless 的核心概念，最绕不过去的是函数生命周期

很多人刚用 Serverless，会把它理解成“函数上传上去就行”。

但平台层真正麻烦的地方，是函数实例的生命周期管理。

从函数被创建、初始化，到进入可用状态，再到闲置和回收，这个过程直接决定了请求延迟、成本和资源利用率。

为什么冷启动总被反复提？

因为这是 Serverless 最典型的体验问题。

所谓冷启动，本质上就是：当没有现成可用实例时，平台得先把执行环境准备出来，然后你的代码才能真正开始跑。

这个准备过程可能包括：

  • 调度资源
  • 创建容器
  • 拉镜像
  • 下载代码
  • 初始化运行时
  • 打通网络
  • 执行用户初始化逻辑

如果这些步骤叠在一起，第一次请求的延迟就会非常明显。

冷启动到底怎么优化？

这事基本可以拆成四层看。

1. 资源调度和容器创建优化

这属于平台层能力。

常见手段包括：

  • 镜像按需加载，避免全量拉取
  • P2P 加速分发镜像
  • 多镜像仓库
  • 预留容器池

2. 代码和依赖下载优化

这部分和用户代码关系更大。

最直观的方法是：

  • 控制代码包大小
  • 尽量压缩依赖
  • 把高频依赖做内置化或层化处理

3. 网络初始化优化

比如打通 VPC 的过程，本身就可能引入明显时延。

4. 运行时和用户初始化优化

这部分经常被忽略，但影响并不小。

比如：

  • 选择合适的运行时
  • 减少初始化时的重操作
  • 对延迟敏感场景使用预留实例
  • 用定时任务提前唤醒实例
  • 对大数据拉取场景合理使用缓存

不同语言的冷启动表现，差异真的很大

这也是很多团队真正落地后才开始有体感的地方。

不同运行时的冷启动表现差异很明显。像 Node.js、Python、PHP 这类语言，一般在函数计算场景中启动更轻；而 Java、.NET 这类运行时，初始化路径往往更重。

这不意味着某种语言不能做 Serverless，而是说：技术栈选择会直接影响你对延迟和成本的权衡。

依赖层为什么重要？

如果多个函数都依赖同一批库，最笨的做法就是每个函数都把依赖打进去。这样会导致：

  • 包越来越大
  • 构建越来越慢
  • 重复存储越来越多
  • 出错概率越来越高

把公共依赖抽成层之后，至少有三个好处：

  1. 函数包更小
  2. 构建过程更干净
  3. 多个函数之间可以复用，减少浪费

触发器，是 Serverless 真正接入业务的入口

很多人以为 Serverless 只是“写个函数然后手动调”。其实在真正的生产场景里，触发器才是函数和外部世界连接起来的关键。

本质上，触发器就是事件驱动的连接器。

它可以来自：

  • HTTP 请求
  • 对象存储事件
  • 消息队列
  • 定时任务
  • 事件总线
  • 云产品事件

从调用方式看，它也可以分成：

  • 同步调用：处理完直接返回结果
  • 异步调用：事件先入队，平台保证可靠处理

运行时不是细节，它会影响整个体验

运行时决定了程序真正执行时所依赖的环境。

你可以把它理解成函数的底座：

  • 用什么语言跑
  • 初始化成本高不高
  • 能不能很好支持现有依赖
  • 性能天花板在哪

这也是为什么同样是 Serverless，不同技术栈做出来的体验会差很多。选择运行时，不能只看团队熟不熟，还得看启动性能、依赖复杂度和业务时延要求。

扩缩容是 Serverless 最核心的系统能力之一

如果说 Serverless 对开发者最有吸引力的地方是“自动扩缩容”，那对平台本身来说，这反而是最难做的部分之一。

一般两类常见思路。

1. Node 维度调度

这种模式更偏平台内部调度逻辑：

  • Scheduler 负责请求路由
  • Local Controller 管理节点上的函数实例生命周期
  • AutoScaler 决定整体扩缩容

平台通过统计节点上闲置和占用状态，来判断是否扩容或回收。

2. Pod 维度扩缩容

在 Kubernetes 生态里，更常见的讨论会落在 Pod 维度。

但普通 HPA 直接套在 Serverless 上其实不完全合适，因为 Serverless 允许缩到 0，而 HPA 传统上依赖已有 Pod 指标来决策。

所以像 Knative 这类体系，会引入自己的扩缩容方案，比如 KPA，用 Revision、Queue Proxy、Activator 等组件来承接从 0 到 1 的启动过程，以及流量统计与扩缩容判断。

流量转发和流量分流，决定了它能不能优雅上线

Serverless 并不只是“函数能跑起来”这么简单。要真正支撑生产，还要解决流量怎么进、怎么分、怎么切版本。

比如在 Knative 这类架构里，Route、Configuration、Revision 这些概念配合起来，才能实现：

  • 新版本发布
  • 流量切换
  • 灰度验证
  • 回滚控制

Serverless 不是什么都好，它也有很现实的限制

Serverless 的限制同样非常明确。

1. 它天然更依赖第三方服务

Serverless 之所以“轻”，很多时候是因为把能力托管出去了。但这也意味着，你对云产品、托管服务和平台 API 的依赖会更重。

2. 对底层硬件控制能力有限

如果你的应用对硬件规格、网络拓扑、GPU/本地盘等有强定制需求，Serverless 可能并不是最优解。

3. 函数间通信效率不一定高

一旦应用被拆得过碎，函数之间的通信、状态管理和排障复杂度会迅速上升。

4. 调试并不总是轻松

本地调试环境、云端运行环境、事件上下文、异步调用链路，这些东西一叠起来，排问题时很容易让人脑壳疼。

5. 不同服务商实现并不统一

Runtime、触发器模型、部署方式、限制项都可能不同，迁移成本并不低。

6. 云厂商绑定是真问题

Serverless 最大的便利之一，往往也是最大的锁定来源。

安全性不能因为“托管”就放松

很多人容易有个错觉：既然 Serverless 是平台托管，那安全就都是云厂商的事。

这想法挺危险。

平台确实接管了底层环境，但你自己的函数逻辑、依赖、权限配置、事件来源校验、第三方调用链，这些风险一样在。

尤其当函数会直接处理用户输入、调用外部资源、或者执行高权限操作时，安全边界不仅没有消失，反而可能因为触发路径变多而更复杂。

最后该怎么理解 Serverless？

Serverless 不是“没有服务器”，而是“尽量别让业务团队围着服务器转”。

它特别适合：

  • 事件驱动场景
  • 波峰波谷明显的业务
  • 快速试错和迭代
  • 希望降低基础设施管理成本的团队

但它也不适合被神化。

一旦业务开始追求极致性能、强控制力、稳定长连接、复杂状态管理或者跨平台一致性，Serverless 的边界就会逐渐显现出来。

所以真正成熟的看法，不是“Serverless 会不会替代传统架构”，而是：

哪些问题应该交给它解决，哪些问题不该硬塞给它。

把这件事想明白，比记住一堆定义有用得多。
