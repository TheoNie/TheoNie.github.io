A2A协议调研

2025-07-30 | AI Infra

近一年，大家都在聊 Agent。

一开始，很多人对 Agent 的理解还比较朴素：无非就是“套了工具调用的大模型”。能查资料、能写东西、能调接口，已经挺像那么回事了。

但真往前走一步，很快就会碰到一个更现实的问题：

如果不是一个 Agent，而是一群 Agent 一起干活呢？

比如：

  • 一个负责和用户对话
  • 一个负责拆解任务
  • 一个负责调用企业系统
  • 一个负责检索资料
  • 一个负责生成最终结果

这时候，问题就不再是“Agent 能不能做事”，而变成了：

它们彼此怎么沟通？怎么协作？怎么知道对方会什么？

A2A，就是在回答这个问题。

A2A 到底是什么？

A2A，全称 Agent2Agent。

名字非常直白：它就是一个让 Agent 和 Agent 之间进行通信与协作 的协议。

你可以把它理解成：给多智能体系统提供一套通用的“交流规则”。

重点不是某个 Agent 内部怎么实现，也不是它背后用的是哪个框架，而是：

  • 它怎么把自己的能力告诉别人
  • 它怎么接收任务
  • 它怎么返回结果
  • 它在任务过程中怎么和别的 Agent 持续交互

说白了，A2A 关心的是：Agent 之间怎么配合干活。

为什么会需要 A2A？

因为只靠一个 Agent，很多时候根本不够。

单个 Agent 做 demo 很容易，真正做复杂业务就没那么轻松了。现实中的任务往往都比较脏、比较长、比较绕：

  • 需要查多个来源的信息
  • 需要跨系统执行动作
  • 需要多轮确认
  • 需要处理中间状态
  • 需要把不同专业能力拼在一起

你当然可以把所有能力都硬塞给一个超级 Agent，但这么干很快就会出问题：

  • 系统越来越重
  • 能力边界越来越糊
  • 调试越来越痛苦
  • 协作越来越像一坨线团

所以更自然的方式，往往是让多个 Agent 分工：

  • 有的负责规划
  • 有的负责执行
  • 有的负责查资料
  • 有的负责跟用户确认
  • 有的负责最后汇总

但一旦多 Agent 出现，新的麻烦也来了：

大家怎么互相理解？

这就是 A2A 想解决的核心问题。

A2A 的设计思路，其实挺务实的

从它的设计原则看，A2A 不是那种喜欢重新发明世界的协议。它的路子反而挺克制。

第一，不要求你暴露内部实现

一个 Agent 不需要把自己的内部逻辑、记忆结构、工具链、推理过程全部公开出来。它只需要告诉外部：

  • 我有什么能力
  • 我怎么接任务
  • 我会返回什么类型的结果

这很符合现实。企业之间协作，不会因为要接协议，就把核心能力全摊在桌面上。A2A 接受这种现实，所以它强调的是能力声明和上下文交换，而不是“彻底透明”。

第二，尽量站在现有标准上做事

A2A 没另起炉灶搞一整套全新基础设施，而是建立在大家已经很熟的东西上，比如：

  • HTTP
  • SSE
  • JSON-RPC

一个协议真正能不能活下来，很多时候不取决于它新不新，而取决于它接起来烦不烦。

第三，它默认任务可能是长时间的

很多任务不是你发一个请求，对面一秒钟吐个结果就结束了。它可能会：

  • 跑很久
  • 处理中间状态
  • 需要补充信息
  • 随时更新进度
  • 最后再给出结果

A2A 明确支持这种长任务、异步优先的模式，所以它不像传统 API 那样只是一次请求、一次返回，而更像一个任务流转过程。

A2A 里最重要的几个角色

A2A 的交互模型里，最核心的是两个角色。

Client Agent

它负责发起任务。简单说，就是“我有个事要做，我来找你帮忙”。

Remote Agent

它负责接任务和执行任务。它可能返回信息，也可能在执行过程中继续向 Client Agent 追问更多上下文。

A2A 最关键的能力：先知道对方会什么

多 Agent 协作里最尴尬的事，就是你想找人干活，但你根本不知道对方会不会。

所以 A2A 很强调一件事：能力发现。

它通过一个叫 Agent Card 的东西，让 Agent 对外描述自己。

你可以把 Agent Card 理解成这个 Agent 的“公开说明书”，里面通常会写：

  • 它是谁
  • 它有哪些技能
  • 它的访问地址是什么
  • 它需要什么认证方式

协议建议把它放在一个固定路径，比如：

/.well-known/agent.json

这样别的 Agent 想找它协作时，先去读这份说明书就行了。

A2A 不是一次调用，而是一整个任务过程

A2A 里还有一个特别重要的概念：Task。

在 A2A 里，任务不是一句消息，而是一个完整的工作单元。它通常会有：

  • 唯一 ID
  • 生命周期
  • 状态变化
  • 输入和输出
  • 中间过程

这就让它天然适合下面这些事情：

  • 跨系统流程
  • 多阶段处理
  • 需要确认的任务
  • 长时运行任务
  • 中间会反复交互的任务

一个典型的 A2A 流程是什么样的？

一次 A2A 协作大致会经历下面几个阶段。

1. Discovery：先发现对方

客户端先通过已知地址获取 Agent Card，看看对方到底会什么、怎么接入。

2. Initiation：正式发起任务

客户端发送任务请求，带上初始消息和任务 ID，开始这次协作。

3. Processing：对方开始处理

Remote Agent 接收任务并执行，执行过程中可能同步返回结果，也可能流式返回，或者异步更新状态。

4. Interaction：中途继续交互

如果任务过程中还缺信息，或者需要用户进一步确认，双方可以继续发送后续消息。

5. Completion：任务结束

任务进入最终状态，比如完成、失败或取消。

为什么大家总把 A2A 和 MCP 放在一起聊？

因为这两个东西，确实很容易被拿来对比。

但它们解决的问题，其实不一样。

MCP 更像“怎么用工具”

MCP 更关注的是模型如何连接外部工具和资源。它强调工具描述、结构化输入输出，以及模型调用工具的方式。

A2A 更像“怎么和别的 Agent 一起干活”

A2A 关注的是 Agent 之间如何发现彼此、发起协作、管理任务、在长流程里持续交互。

工具不是 Agent，这点别混了

工具通常具备这些特点：

  • 输入输出清晰
  • 功能边界明确
  • 一次调用完成一次动作
  • 通常比较无状态
  • 结果比较可预测

而 Agent 更像“带自主性的协作者”。它不只是执行指令，还可能会：

  • 分析问题
  • 制定计划
  • 调用多个工具
  • 持续维护上下文
  • 根据中间结果改变策略
  • 在多轮对话中推进任务

所以，工具更像零件，Agent 更像角色。MCP 很适合描述“怎么使用零件”，A2A 更适合描述“角色之间怎么配合”。

A2A 和 MCP，最合理的姿势其实是一起用

很多人喜欢问：A2A 和 MCP 谁更重要？

更现实的答案是：一起用。

一种很自然的架构是：

  • Agent 对外通过 A2A 和其他 Agent 协作
  • Agent 对内通过 MCP 去调用工具和资源

也就是说：

  • A2A 负责“对外协作”
  • MCP 负责“对内调用”

最后总结一下

如果你只想记住一句话，那就是：

A2A 解决的是 Agent 和 Agent 之间怎么协作。

它关注的是：

  • 能力发现
  • 任务管理
  • 多轮交互
  • 长时任务
  • 上下文交换
  • 结果输出

而不是单纯的工具调用。

如果说 MCP 是在解决“模型怎么接工具”，那 A2A 更像是在解决“当系统里有很多智能体时，它们怎么彼此配合”。

这件事现在还在早期，但方向很明确。

Agent 如果真的要走进企业、走进复杂业务、走进长期流程，最后一定绕不开一个问题：

它们不能只会自己干活，还得会一起干活。
