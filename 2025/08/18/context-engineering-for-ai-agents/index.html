为什么 AI Agent 不是上下文越长越强：从 Manus 实践看上下文工程

2025-08-18 | AI Engineering

这两年，大模型的上下文窗口越做越大，很多人也自然形成了一种直觉：窗口越大，Agent 就越聪明，系统就越稳定。

但真把 Agent 放进复杂任务里跑一段时间，你很快会发现，事情根本不是这么回事。

上下文变长，确实意味着模型能“看到”的东西更多；但与此同时，噪音也更多，冲突也更多，注意力也更容易跑偏。一个没有经过管理的上下文，并不会自动变成优势，反而很容易把系统拖进一种高成本、低质量、难调试的状态。

这也是为什么“上下文工程”（Context Engineering）正在变得越来越重要。

如果说提示工程解决的是“怎么把一句话说清楚”，那上下文工程解决的就是另一层问题：在一个持续运行、会调用工具、会执行长链任务的 Agent 系统里，应该把什么信息留在上下文里，又该把什么东西移出去。

最近读完《How to Fix Your Context》以及 Manus 团队关于上下文工程的实践总结后，一个很明确的感受是：上下文的核心问题，从来不是塞得不够多，而是管得不够好。

上下文真正的问题，不是容量，而是失控

很多人一提到上下文问题，第一反应都是“窗口不够大”。

但从工程角度看，这往往只是表象。真正更棘手的问题其实是：随着上下文变长，模型的注意力开始失效。

它可能记得很多东西，却不再能稳定地抓住真正重要的东西。于是系统开始出现一系列表面不同、实则同源的问题。

1. 上下文污染

这是最危险的一种情况。

如果模型在某一轮推理里产生了幻觉，或者做出了错误判断，而这些错误又被继续保留在上下文里，后面的轮次就可能把这些错误当成既成事实反复引用。久而久之，整个推理链条会在错误前提上不断自我加固。

这类问题最麻烦的地方在于，它不是一次性报错，而是会像把一滴墨水滴进清水里一样，慢慢把整个上下文染脏。

2. 上下文干扰

当上下文里堆了太多历史记录、工具定义、检索结果和中间过程时，模型并不会自动把重点挑出来。

现实往往正相反：它会被大量近似相关的信息吸走注意力，真正关键的目标和约束反而被淹没。很多长任务失败，不是因为模型“不会”，而是因为它已经被上下文拖偏了。

3. 上下文混淆

混淆通常来自“看起来都有关，但其实没必要一起出现”的信息。

最典型的就是工具配置。工具一多，描述一重叠，模型就会开始犹豫：到底该调用哪个？如果工具数量继续膨胀，最终结果往往不是能力更强，而是选择更乱。

4. 上下文冲突

长流程任务中，信息是分阶段进入系统的。问题在于，前面阶段生成的不完整推理、试探性回答、错误判断，常常会和后面拿到的更完整信息发生冲突。

一旦这些“历史包袱”都留在同一条上下文里，模型就很容易在多个互相矛盾的信号之间来回摇摆，最后输出一个不够稳定的结果。

别把上下文当仓库，更应该把它当运行时环境

上下文不是一个被动的容器，不是“把信息往里堆”这么简单。它更像是一个运行时环境：

  • 它决定模型此刻能看到什么
  • 它影响模型把注意力放到哪里
  • 它会塑造模型的行为路径
  • 它甚至会决定模型接下来犯什么错

换句话说，上下文管理不是文档整理，而是系统设计。

这也是为什么真正有效的上下文工程，关注点不会停留在“能不能塞更多 token”，而是会转向几个更现实的问题：

  • 哪些信息必须进主上下文？
  • 哪些信息应该外置？
  • 哪些历史应该保留，哪些应该裁剪？
  • 模型此刻最应该关注什么？

Manus 给出的几个很硬的启发

Manus 的实践之所以值得看，不是因为它讲了很多新概念，而是因为它把一些容易被忽略的工程事实讲明白了。

第一，稳定结构比灵活切换更重要

直觉上，很多人会觉得工具配置应该尽量动态：任务变了，就删掉旧工具、换上新工具，看起来非常灵活。

但问题是，大多数模型的 KV Cache 都依赖上下文前缀的稳定性。一旦你频繁改动工具定义，缓存就会失效，成本和延迟立刻上来，模型行为也更容易漂。

Manus 的思路更务实：结构尽量保持不变，行为再做动态控制。

也就是说，不一定要通过修改 Prompt 告诉模型“你现在只能用这些工具”，而可以保留一套稳定的工具全集，再通过掩码、预填充等方式限制当前可用范围。

这套思路的关键，不只是节省 token，更重要的是让系统在长流程任务里保持可预测。

第二，错误不是垃圾，而是高价值信号

很多系统一发现模型走偏，就本能地想“把错的删掉”。

但从 Agent 设计角度看，错误往往不是应该被清空的噪音，而是系统修正自身的重要依据。你如果把失败轨迹全部擦掉，相当于把模型曾经走错的证据也一并删了。

更合理的做法不是让错误继续污染主上下文，而是把这些失败记录卸载到外部存储，在需要时作为总结和反思的材料重新引用。

这就引出了另一个关键策略：上下文卸载。

第三，长任务几乎一定要依赖外部记忆

只靠主上下文硬扛长流程任务，通常撑不久。

无论是 Anthropic 提出的 think/scratchpad 思路，还是 Manus 把文件系统当作长期工作空间，本质上都在做同一件事：把中间状态、失败记录、文档内容、分析过程这些东西移出主上下文。

主上下文负责即时决策，外部存储负责长期承载。两者分工明确，系统才容易稳定。

这件事非常像人在做复杂工作时的习惯：脑子里只保留当前最重要的判断，其他内容写到便签、文档、任务板里。真正高效的系统，不是把一切都记在脑子里，而是知道什么该留在脑子里。

第四，任务清单本身就是注意力控制器

Manus 有个很有意思的实践：用一个简单的 todo.md 来维持长任务流程。

表面上看，这只是任务管理；但从上下文工程角度看，它其实是在不断把当前目标、状态和下一步动作重写到上下文末尾。由于模型天然对近期内容更敏感，这种重复复述会显著降低目标漂移的概率。

说白了，待办清单不只是给人看的，也是在给模型“拉注意力”。

真正有用的上下文管理策略

压缩成工程视角，可以总结成六件事。

1. RAG 的重点不是多，而是准

RAG 最容易踩的坑，就是“查到了就全塞进去”。

但真正有效的 RAG，从来不是信息搬运，而是语义增强。你不是要把所有相关资料都倒给模型，而是要把当前任务最需要的那一部分，用模型最容易吸收的方式送进去。

更进一步的做法，是不要把完整内容都塞进上下文，而只保留：

  • 信息路径
  • 摘要
  • 获取方式

这样模型知道“信息在哪里”，需要时再取，而不是一开始就被细节淹没。

2. 工具不是越多越强，越多越容易乱

给 Agent 加工具当然是必要的，但工具集合不能失控。

如果工具名相似、描述重叠、可选项太多，模型会把大量注意力耗在“到底该选哪个”上。结果就是能力看似丰富，实际成功率反而下降。

所以工具管理的原则不是堆满，而是收敛：

  • 只暴露当前阶段真正相关的工具能力
  • 工具命名保持清晰、一致、可分组
  • 尽量让结构稳定，别反复改前缀和定义

3. 复杂任务要拆线程，而不是堆一锅

一个大任务里经常混着多个子问题：查资料、写分析、调接口、做验证、整理结论。

如果所有过程都塞进一条上下文里，冲突和混淆几乎不可避免。更稳的方法，是把任务拆成多个逻辑线程，让不同子任务在各自独立的上下文里运行。

这也是多智能体、多阶段流水线设计的一个根本价值：不是为了炫技，而是为了隔离噪音。

4. 给模型准备一个外部便笺簿

这是最值得尽快落地的一条。

无论是文件系统、数据库、临时工作区，还是一个 scratchpad 工具，本质上都在帮助模型把不该长期占据主上下文的内容挪出去。

比如：

  • 原始网页内容
  • 长文档全文
  • 中间分析过程
  • 错误日志
  • 多轮尝试记录

这些东西都可以外置。模型只需要在当前轮次里引用必要摘要，而不是把整堆历史一直背在身上跑。

5. 修剪和总结的目标，是重新分配注意力

很多人把 summary 当成“压缩 token”的技巧，其实这只是表层收益。

更关键的是，总结能帮助模型重新聚焦：

  • 当前目标是什么
  • 已完成了什么
  • 还剩什么没做
  • 哪些历史结论仍然有效

高质量总结不是缩写版聊天记录，而是一次主动的注意力重排。

6. 少样本示例也会把模型带进沟里

模型很擅长模仿。问题是，它有时候模仿得太认真了。

如果你在上下文里反复给出相似结构、相似动作、相似节奏，模型很容易进入惯性，机械重复之前的路径，而不再根据当前状态做调整。

所以在重复性任务里，保留少量受控变化反而是好事。适度打破节奏，能让模型从“照着演”切回“重新判断”。

对做 Agent 的人来说，最重要的心态转变是什么？

别再把上下文当成越大越好的缓存池，而要把它当成需要精心维护的运行时。

这意味着你在设计 Agent 时，需要思考的重点不再只是：

  • 模型够不够强
  • 窗口够不够大
  • 提示词写得漂不漂亮

而是：

  • 当前轮次真正需要什么信息
  • 哪些信息应该外置
  • 历史错误怎么保留但不污染主线程
  • 如何让目标始终停留在模型近期注意力里
  • 如何让系统结构稳定，不因为动态变化把自己搞崩

结语

上下文工程听起来像一个新词，但它解决的问题一点都不新。

凡是做过复杂系统的人，最后都会遇到类似的挑战：信息越来越多，状态越来越复杂，历史越来越长，系统开始变得不稳定。AI Agent 只是把这件事在大模型时代重新演了一遍。

区别在于，传统软件的状态管理失控，会带来 bug；而 Agent 的上下文管理失控，带来的往往是更隐蔽的东西：幻觉、漂移、反复横跳、工具误用、以及你明明看不出哪里错了但结果就是不对。

所以，上下文工程本质上不是“提示技巧”的升级版，而是 Agent 系统设计的一部分。

真正稳定的 Agent，不是因为它记住了所有东西，而是因为它知道：什么该记，什么该忘，什么该移出去，什么该被再次强调。
